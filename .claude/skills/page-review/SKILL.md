# 内容页评审标准

## 使命

页面写给零基础学习者，页面上每个数字、每条推导都可能变成读者的投资判断依据。评审的使命是替读者提前踩雷：读者分辨不出页面哪里错了，评审员能。

评审与生产分离：只产报告，不改页面。改什么、怎么改，用户看报告后拍板。

## 评审模式（动手前先定）

同一份 skill 覆盖两种模式，跑哪些层取决于页面处于什么阶段：

- **Mode A · 首审**（新页面第一次评审，或从未跑过评审的页面）：跑 **§A 数据/信息层**（准确性 + 严谨性 + 完整性）+ **§B 核心结论层**（骨架卡专属）。目标是把页面的数字、来源、结论一次审到位。
- **Mode B · 已修卡二审**（页面已经跑过一轮数字/信息评审并按报告修复过、现在做第二轮）：**跳过 §A 数据/信息层**（那一轮已经做过、不再重复浪费），**只跑 §B 核心结论层 + §C 改动影响面追溯层**。目标是补第一轮的两个盲区——核心结论本身立不立得住、上一轮修复留没留下影响面漏洞。

两种模式共用的部分：产出格式、修复流程要求、多 agent 组织。任务书里必须明确写清是 Mode A 还是 Mode B，评审员按模式跑相应层，不越界不漏项。

## 怎么评

**从怀疑出发**：假设页面有错，去找证据；不是假设页面对，去做确认。生产页面的也是 AI，高发错误形态：凭记忆写数字、来源与断言张冠李戴、推导读着流畅实则跳步、教学简化被当成事实陈述。

### §A 数据/信息层（Mode A 首审必跑；Mode B 二审跳过）

两个核心角度：

1. **准确性**——数字、事实、来源真不真。关键断言要回源核验：先打开徽章指向的来源（WebFetch）对数字和口径；打不开或对不上，WebSearch 找一手来源。核验结论只有三种老实写法：**对上了／对不上（附证据）／无法核验（说明查了什么）**，不把「没查到反例」写成「已核验」。优先警惕两类隐蔽错误：**方向对但数字错**（记忆污染）、**数字对但口径错**（年份、币种、财年与自然年、统计范围、约当单位）。写作时正确、现在已过时的数字标「已过时」，与「错误」分开——处置方式不同。
2. **严谨性**——页面上所有内容、信息、结论都必须有可靠来源支撑；推论只在标注成推论的前提下允许存在。评审抓三种违规：**无来源断言**（一条结论查不到支撑、页面也没标注它是推论）、**推论冒充事实**（推出来的结论写得像来源里的原话，含徽章分类错误——该标估算/专家判断的标成已核验、教学简化没标注）、**来源不够格**（媒体转述、二手汇编被当一手来源用）。推论本身的逻辑对不对，也要正常客观评审，但它不是最核心的：有明显问题就报，不明显不重要的可以忽略。

来源分两类，评审时分别看：

- **硬事实来源**（一手资料：公司财报与官方资料、10-K/8-K、电话会转录、官方统计、可靠行业研究与调研）——用来核实数字、事件、公司具体表态是否属实。断言型内容必须有硬事实来源支撑，否则算无来源断言。
- **高质量讨论文章**（观点讨论来源：分析师深度报告、学术论文、权威媒体深度调查、圈内知名投资人的公开表述/股东信/演讲/访谈，具有独立论证过程、非转述摘要）——用来把结论/观点与观点市场连起来。骨架卡的核心结论必须挂至少 2 篇这类文章；页内其他次级结论按需挂。评审时区分标准：这篇文章是在**报道事实**（属硬事实来源）还是在**讨论观点**（属高质量讨论文章）——同一篇文章的不同段落可能分属两类。

一个次级角度：

3. **完整性**——只报明显缺失：本页承载的知识范围内缺了一块导致读不通整篇、机制链断在半路、例子讲了一半。可补可不补的不报。前置知识属于其他卡或模块，不算本页缺失。

### §B 核心结论层（Mode A + Mode B 通用 · 骨架卡专属）

**骨架卡多审一层：卡的核心结论本身**。每张骨架卡承载一个模型，整张卡的落点就是一个核心结论。除了逐条审页内内容，还要跳出来，从第一性原理审这个核心结论本身，六个维度：

1. **准确性**——结论描述的现象/机制在物理、产业、商业逻辑上真的成立，还是几个案例归纳出来的表象规律。
2. **漏洞**——从前提到结论的推理链条有没有偷换概念、以偏概全、跳步、把相关性冒充因果。
3. **缺失**——结论表述里有没有省略的重要条件、隐含前提，读者按字面理解就会用错。
4. **边界**——卡里划的边界是不是真边界，有没有明显反例场景没被圈进边界里；边界表述有没有过度绝对化或过度模糊。
5. **结论出处可靠性**——结论的推出过程是从可靠一手材料 + 严谨推理来的，还是二手转述、圈内口耳相传、少数案例归纳来的。
6. **高质量讨论文章**——观点市场上有没有讨论过这个结论的高质量文章（见严谨性段的定义），页面来源清单里有没有挂——**这一层不是找证据支持结论，是把结论与观点讨论环境连起来**：观点在市场上有没有被系统讨论过，读者点开一篇文章就能看到别的深度声音，是判断结论成熟度的一个指标。

核心结论本身立不住，页内每个数字都对也救不回来。

### §C 改动影响面追溯层（Mode B 二审专属）

对已经修过一轮的页面做二审时，除了 §B 核心结论层，必须额外审一层「上一轮修复是否留下影响面漏洞」。上一轮修复动作只把数据/信息从 X 换成 X' 是最容易漏的失败模式——**数据换了但论证方向、结论表述、逻辑链条、边界段没跟着改**。执行动作：拉页面的 git 修改历史（`git log --follow output/spineNN.html`），对每一处已经改过的位置逐一审：改的是什么（老值→新值）／这个数据/信息原本撑起来的论证段落是什么、结论表达是什么、边界段有没有依赖／改动后论证方向、结论表述、逻辑链条、边界是否还成立／不成立时具体要跟着改哪句、软化哪个断言、补哪条边界。**发现的漏洞按常规问题清单格式报**，标出"改动影响面漏洞 · 上一轮 commit 号"。首审新页面不适用这一层。

角度之外，任何会误导投资判断的问题照报。

**拿不准也要报**，标「存疑」，主协调者会复核——正确的模糊好过精确的错误，别自己吞掉。

**基准是项目立的标准，不是个人口味**：页面的验收标准在它对应的生产 skill 里（骨架卡 `.claude/skills/spine-card/SKILL.md`，手册页 `.claude/skills/manual-page/SKILL.md`），与标准不一致才算问题，「换我会另一种写法」不算。

## 产出格式

每页一份清单，每条问题包含：

- **严重度**：高＝内容错了（事实错误、数字与来源对不上、结论与可靠来源相悖）／中＝内容未必错但纪律违规（断言无来源支撑、推论未标注成推论、简化未标注、来源不够格；完整性的明显缺失）
- **位置**：小节名＋原文片段引用，不打开源码也能定位
- **问题**：一句话说清错在哪
- **证据**：核验过程与来源（准确性类问题必须有）
- **联动位置**：该问题涉及/牵动的**所有其他位置**——相邻论证段落、SVG/JS 数据点、练习题前提、边界段、来源清单、图表标签等。一个问题往往不只错在一处：一个数字改动可能牵动 SVG 数据 + 练习题前提 + 相邻推导链条 + 页脚来源，评审员的责任是列全**联动清单**，不是只报单点。修一处漏了联动 = 数据换了但论证/图/前提没换 = 与不修等价甚至更差（因为看着像修过）。
- **置信**：已确认／存疑待复核

清单后两行整页判断：一句话总评；该页在准确性与严谨性上是否达到其生产 skill 的验收线。

**禁止交付**「立得住 / 无硬伤 / 合理切割 / 基本合规」这类总结式评语——出现打回重派。评审员不能靠"没找到硬伤 → 立得住"的默认推理，要主动构造反例场景来试；每条发现必须给具体位置与证据，不能只给总结。

## 修复流程要求（主协调者执行）

评审员给出的**联动位置清单**是修复的操作单，不是参考。修复流程硬要求：

1. **每修一处，先过一遍该条的联动清单**——把评审员列出的所有联动位置逐一检查，看是否需要一起改。漏改联动 = 数据换了但论证/图/前提没跟着换 = 页面进入"看着修过实际不一致"的坏状态。
2. **改数据必须重新审视论证方向**：某些数据改动会翻转原论证的方向（如"14.1→0.6 万亿是全公司同比"改成"DS 部门 0.27→-4.58 万亿环比"，跨期从 12 月变 3 月——原论证链条是否还成立？是否反而更强？还是变弱？必须显式判断，不能只替换数字）。
3. **改来源硬度必须重新审视结论表述**：某个数字从"已核验"降级到"专家判断/传闻口径"，依赖该数字的论证/推演题前提/结论口号都要跟着软化，不能只降徽章不改文本。
4. **提交前一致性 spot check**：grep 页面里的关键数字/术语/时点，确认全页多处出现的地方都改到位（SVG JS 数据、正文、图表标签、练习题、来源清单、页脚说明），不留只改一半的痕迹。

## 多 agent 组织（主协调者执行）

- 同一页由 **2 个（或以上）评审 agent 独立评审**，都用 **opus 模型**，互相不知道对方存在、不看对方结论。独立是防遗漏的手段：一个评审员的盲区靠另一个的不同切入补，两份清单的并集才是这一页的发现面。
- **两名评审员的分工按模式定**：
  - **Mode A · 首审**：两名评审员**都跑 §A + §B**（骨架卡的话），完全独立，靠切入角度不同互补。
  - **Mode B · 已修卡二审**：两名评审员**分工不同层**——评审员 1 跑 §B 核心结论 6 维度审（含高质量讨论文章搜集），评审员 2 跑 §C 改动影响面追溯。任务书里明确写清各自跑哪一层。
- 合并清单后做**对抗复核**：两人不一致的发现（一方报了另一方没报、或结论相反），以及全部高严重度发现，派复核 agent 带着具体分歧去证伪——复核的姿态是**推翻它**，推不翻才站住。错误的指控与页面本身的错误同样有害。**复核高严重度指控时，指控里引用的具体数字与口径必须一手核验**（打开来源对数），不能默认评审员引对了——评审员自己的数字也会错（先例：spine06 二审，评审员把营收口径数字当价格口径，报出错误的 81%）。
- **专项复核姿态反向**：涉及"数据联动漏洞"的复核（评审员宣称当时的修改漏改了某处联动）姿态是**证明当时修改已经改到位**，证不了才站住；涉及"模型层反例"的复核（评审员构造反例场景断言卡的边界没圈到）姿态是**证明边界段已切住这个反例**，切不住才站住。这两种姿态与常规"推翻指控"的姿态方向一致（都是"给页面辩护"），但对象不同。
- 所有 agent 只读不写，任务书明确禁改任何项目文件；任务书薄：指到本 skill、对应生产 skill、页面路径即可，判断空间留给评审员。
- 每批 1 页（2026-07-14 用户判定，从 2 页改）。复核后去重，按页面 × 严重度直接在对话里列给用户，不写报告文件。
